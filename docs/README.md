# 工作流系统使用指南

## 📚 概述

本项目为 Y1000 MMORPG 逆向重写项目配置了智能工作流自动匹配系统。根据任务类型，AI 将自动选择合适的工作流程。

## 🔄 可用工作流

### 1. 分析类工作流 (`/analysis-worker`)
**适用于**：项目拓扑分析、协议逆向、业务逻辑提取、数据库逆向

**触发关键词**：分析、逆向、提取、识别、理解、调研

**典型任务**：
```
"分析 LoginServer 的网络协议"
"提取战斗系统的业务逻辑"
"逆向数据库表结构"
```

### 2. 设计类工作流 (`/design-worker`)
**适用于**：架构设计、系统拆分、技术栈选型

**触发关键词**：设计、架构、规划、方案、选型

**典型任务**：
```
"设计新的用户认证架构"
"规划服务器拆分方案"
"选择技术栈"
```

### 3. 实现类工作流 (`/implementation-worker`)
**适用于**：功能开发、模块实现、系统构建

**触发关键词**：实现、开发、编码、重写、创建

**典型任务**：
```
"实现角色创建功能"
"开发战斗系统"
"构建网络通信模块"
```

### 4. 验证类工作流 (`/verification-worker`)
**适用于**：单元测试、功能测试、性能测试

**触发关键词**：测试、验证、对比、检查、benchmark

**典型任务**：
```
"测试登录系统"
"验证战斗伤害计算"
"性能基准测试"
```

### 5. 修复类工作流 (`/fix-worker`)
**适用于**：Bug修复、性能优化、代码重构

**触发关键词**：修复、调试、优化、重构、bug、问题

**典型任务**：
```
"修复连接池泄漏问题"
"优化玩家查询性能"
"重构数据访问层"
```

## 🎯 使用方式

### 自动匹配（推荐）
直接描述任务，AI 会根据关键词自动选择工作流：

```
"分析 GameServer 的玩家系统"  → 自动使用 analysis-worker
"设计物品系统架构"           → 自动使用 design-worker
"实现背包功能"               → 自动使用 implementation-worker
```

### 显式指定
使用斜杠命令强制使用特定工作流：

```
/analysis-worker 分析登录流程
/design-worker 设计缓存方案
/implementation-worker 实现交易系统
/verification-worker 测试战斗系统
/fix-worker 修复内存泄漏
```

## 📋 工作流程示例

### 完整重写流程
```
1. /analysis-worker 分析项目拓扑
   → 输出：00_ProjectTopology.md

2. /analysis-worker 逆向网络协议
   → 输出：01_NetworkProtocol.md

3. /analysis-worker 提取游戏业务逻辑
   → 输出：02_GameDesign.md

4. /analysis-worker 逆向数据库设计
   → 输出：03_DatabaseDesign.md

5. /design-worker 设计现代架构
   → 输出：04_TechnicalArchitecture.md
   → 等待用户批准

6. /implementation-worker 实现核心框架
   → 输出：src/core/*

7. /verification-worker 测试核心框架
   → 输出：verification_report.md

8. /implementation-worker 实现认证系统
   → 输出：src/game/auth/*

9. /verification-worker 测试认证系统
   → 继续下一系统...
```

## ⚙️ 工作流特性

### 检查点机制
每个工作流在关键节点都会暂停，等待用户确认：

```
🛑 检查点 X：【阶段名称】完成

【关键发现】
...

【用户选项】
1. 输入"继续" - 下一阶段
2. 输入"修正 XXX" - 调整结果
3. 输入"暂停" - 保存进度
```

### 双层输出
每个阶段生成两类文档：
- **Markdown**：人类可读的详细分析
- **JSON**：机器可读的结构化数据

### 自动化工具调用
工作流中标记 `// turbo` 的命令会自动执行，无需用户批准

## 🚫 约束系统

### 红线（禁止）
- ❌ 复制旧代码超过 5 行
- ❌ 使用全局变量
- ❌ 明文存储密码
- ❌ 硬编码配置

### 黄线（需说明）
- ⚠️ 使用过程式代码
- ⚠️ 手写 SQL
- ⚠️ 违反 DRY 原则

### 最佳实践
- ✅ 使用泛型集合
- ✅ 使用接口
- ✅ 使用异常处理
- ✅ 配置文件管理

## 🔍 常见问题

### Q: 如何选择合适的工作流？
A: 通常按以下顺序：分析 → 设计 → 实现 → 验证 → 修复（如需要）

### Q: 可以跳过某些阶段吗？
A: 不推荐。特别是"设计"阶段必须完成并获批准才能进入实现。

### Q: 遇到问题如何处理？
A: AI 会自动暂停并请求决策，提供多个方案供选择。

### Q: 如何恢复中断的任务？
A: 使用命令"继续上次任务"即可恢复。

## 🎓 最佳实践

1. **遵循顺序**：不要跳过分析直接实现
2. **小步迭代**：每次完成一个小系统
3. **充分测试**：实现后立即验证
4. **记录决策**：重要决策要有 ADR 记录
5. **管理技术债务**：用注释标记临时方案
