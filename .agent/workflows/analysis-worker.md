# 分析类任务工作流

## 🎯 适用场景

- 项目拓扑分析（阶段 0）
- 网络协议逆向（阶段 1）
- 游戏业务逻辑提取（阶段 2）
- 数据库逆向（阶段 3）

## 🔑 触发关键词

`分析`、`逆向`、`提取`、`识别`、`理解`、`调研`、`扫描`

## 📋 执行步骤

### 1. 明确分析目标
- 确认需要分析的具体内容（协议/逻辑/数据库）
- 确定分析范围和优先级
- 识别关键文件和模块

### 2. 自主考古（禁止询问用户要文档）
// turbo
- 使用 `find_by_name` 扫描相关文件（`.dpr`, `.pas`, `.dfm`）
- 使用 `grep_search` 搜索关键常量和类型
  - 协议：`packed record`, `CM_*`, `SM_*`
  - 数据库：`SELECT`, `INSERT`, `UPDATE`, `CREATE TABLE`
  - 业务逻辑：状态变量、核心函数名
- 使用 `view_file_outline` 快速浏览文件结构
- 使用 `view_code_item` 深入分析关键代码段

### 3. 绘制拓扑/结构图
- 使用 Mermaid 图表可视化
  - 架构图（服务器/模块关系）
  - 协议流程图
  - 状态机图
  - ER 图（数据库）

### 4. 提取核心规格
- 创建结构化数据（JSON 格式）
  - 消息 ID 映射表
  - 数值公式
  - 表结构定义
- 标注原始代码位置（文件名 + 行号）

### 5. 识别问题和歧义
- 发现的问题分类：
  - 🔴 确定的 Bug
  - 🟡 疑似 Bug（需决策）
  - 🟢 设计缺陷
  - 🔵 安全漏洞
- 遇到以下情况**必须暂停**：
  - 多个互相矛盾的实现
  - 疑似 bug 但无法确定
  - 缺少关键信息

### 6. 生成双层输出

#### 第一层：Markdown 文档
- 必须有目录
- 必须有关键发现摘要
- 必须有图表（Mermaid）
- 必须标注原始代码位置
- 必须包含问题清单

#### 第二层：JSON 数据
- 供后续阶段快速检索
- 供自动化工具处理

### 7. 提交检查点
按照以下格式提交：

```
🛑 检查点 X：【XXX分析】完成

【关键发现】
- 发现项 1
- 发现项 2

【输出文档】
- 📄 XX_YYY.md
- 📄 XX_YYY.json

【待决策问题】
- 问题 1
- 问题 2

【下一步】
进入【阶段 X+1】 / 继续当前阶段

【用户选项】
1. 输入"继续" - 下一阶段
2. 输入"修正 XXX" - 调整结果
3. 输入"暂停" - 保存进度
```

## ⚠️ 注意事项

1. **避免完美主义**：不要试图理解每行旧代码，60% 可能是无用代码
2. **上下文管理**：文件超过 10,000 行时拒绝全文读取，使用搜索和轮廓功能
3. **不确定时暂停**：发现歧义、bug、安全漏洞时必须请求决策
4. **标记技术债务**：用注释标记待优化项

## ✅ 输出检查清单

- [ ] Markdown 文档完整（含目录、图表、代码位置）
- [ ] JSON 数据结构清晰
- [ ] 问题分类明确
- [ ] 原始代码位置已标注
- [ ] 图表清晰可读
- [ ] 检查点格式正确
